# OneWish OS — Full List: Settings, Operating Strategies, Integrations & Principles for Cursor, Desktop, Cloud & Agents

**Purpose:** Single reference for **everything you should use, configure, and/or set** for **Cursor** (desktop and cloud), **agents**, and **agent systems** so they align with OneWish OS. Covers settings, operating strategies, integrations, principles, and checklists.

**Governance:** [constitution.md](constitution.md); [DEV_TOOLS_RULES_SETUP.md](DEV_TOOLS_RULES_SETUP.md); [DEV_TOOLS_CANONICAL_LIST_AND_AUDIT.md](DEV_TOOLS_CANONICAL_LIST_AND_AUDIT.md); [NEVER_MISS_INSTRUCTIONS.md](NEVER_MISS_INSTRUCTIONS.md); [AGENTS.md](../AGENTS.md).

## 1. Principles (apply in all tools and agents)

| Principle | Source | What to do / set |
|-----------|--------|------------------|
| **Human-in-the-loop** | Constitution §1 | Never auto-approve privilege, filings, productions, expert narratives, damages conclusions, compliance. Route high-stakes outputs through human review. |
| **Claim typing** | Constitution §2 | Every assertion tagged FACT / INFERENCE / OPINION / LEGAL_CONCLUSION. FACT requires evidence pointer or UNVERIFIED. |
| **Explainability & audit** | Constitution §3 | Preserve prompts, tool calls, parameters, outputs; immutable audit trail; processing history (stage, agent, timestamp, hashes). |
| **Bias & safety** | Constitution §4 | Evaluate for bias and harmful outputs; document mitigations; attorney review for sensitive inferences. |
| **Simulation disclaimer** | Constitution §5 | All litigation outputs are drafts; not legal advice; attorney review required. Include disclaimer in generated docs. |
| **Agents execute (default)** | Constitution §6 | Agents run via terminal/SDKs/IDEs/CLIs — not the user. Do not ask the user to run commands the agent can run. |
| **Outcome + chart/graph** | Constitution §6.5 | At each stage: provide example of outcome and derivative outcomes; provide chart or graph (or reference). |
| **New capabilities** | Constitution §7 | Read constitution, my-agent.agent.md, master-project-rules.mdc first; create skill; update indexes; follow NEW_CAPABILITY_CHECKLIST. |
| **Dev tools same level** | Constitution §8.1 | Cursor, Claude Code, Copilot, Codex, v0, VS Code, Windsurf, Warp, Goose — same governance, entry points, ability to run any OneWish OS script. |
| **Cross-tool execution** | Constitution §8.5 | From any connected dev tool, run any OneWish OS script from master repo terminal (e.g. run-audit-runbook.sh, connect-and-configure-onewishos.sh). |
| **Never-miss propagation** | Protocol §5 | Route and sync so no target is skipped; run route-constitution and sync-all-repos per triggers; complete Full Review mandatory checklist. |
| **Rigid review** | Protocol | Every change touching protocol scope must pass rigid review: usability, operability, efficiency. |
| **Iron curtain** | Constitution; AGENT_SKILLS_KNOWLEDGE_TOOLS_CADENCE | All updates (knowledge, memory, artifacts, rules) automated multi-laterally; do not disrupt cadence across platforms. |
| **Single source of truth** | NEVER_MISS | Master repo = GitHub (CURSOR CLOUD AGENTS); canonical files per category (repos → config/repos.yaml; MCPs → canonical-mcp-connectors.yaml). |
| **Product that builds the product** | PRODUCT_THAT_BUILDS_THE_PRODUCT_AND_ARCHITECTURE_TENET | Architecture = core tenet; after each review: commands organized in place, architecture updated/reviewed per CS/repos/best practices. |

## 3. Operating strategies (when and how to run)

### 3.1 Never-miss sequence (order)

Run in this order from **any** dev tool (Cursor, Claude Code, Codex) with master repo or workspace open:

```
1. (If MCP/connector list changed) Edit config/canonical-mcp-connectors.yaml only → python3 scripts/sync-canonical-mcp-connectors.py
2. Route governance: ./scripts/route-constitution-to-repos.sh
3. Sync all repos: ./scripts/sync-all-repos-and-review.sh
4. Refresh workspace: ./scripts/sync-dev-tools-repos-from-config.sh
5. (Optional) Verify connections: ./scripts/verify-all-connections.sh
6. Full Review: ./scripts/run-full-review.sh
7. Complete mandatory checklist: docs/audits/full-review-summary.md
8. Commit and push master; then child repos if they have remotes.
```

**Shortcut:** Steps 2–6 in one go: `./scripts/connect-and-configure-onewishos.sh` (then 7 and 8).

### 3.2 When to run what

| Trigger | What to run / set |
|---------|-------------------|
| **After governance or connector change** | Never-miss sequence (at least route + sync). |
| **Before commit (any change in master)** | Full Review + mandatory checklist; then commit; then push master. |
| **Weekly / recurring** | sync-all-repos-and-review.sh; optionally Full Review. |
| **Quarterly** | Full Review; omni audit (run-omni-audit-and-never-miss.sh); complete checklist; apply recommendations per policy. |
| **After adding repo** | Add to config/repos.yaml; sync-dev-tools-repos-from-config.sh; route + sync. |
| **After adding MCP/connector** | Edit canonical-mcp-connectors.yaml only → sync script → connect-and-configure-onewishos.sh → commit → push. |
| **Full Review (all types)** | ./scripts/run-review.sh (runs run-every-turn: Go → codify → no-fail Comprehensive Review). |

### 3.3 Single point of truth (where to commit)

| Content | Commit to |
|---------|-----------|
| Governance, platform config, scripts, docs, .cursor, .github | **Master repo** (CURSOR CLOUD AGENTS). |
| Project-specific code/docs owned by a child repo | That child repo (e.g. LitigationForce.AI). |
| **Then** | Push master to GitHub; run route + sync so children receive updates. |

## 6. Checklist: confirm/set for Cursor desktop, cloud, and agents

Use this to verify or set everything in one pass.

### 6.1 Cursor app and workspace

- [ ] Master repo (CURSOR CLOUD AGENTS) opened as folder or part of multi-root workspace.
- [ ] Multi-root workspace includes master and any child repos you work on (or use onewish-repos.code-workspace).
- [ ] .cursor/rules/master-project-rules.mdc exists and has alwaysApply: true.
- [ ] .cursor/rules/platform-bootstrap.mdc exists (and in children via sync).
- [ ] AGENTS.md in root references constitution, checklist, master rules, Full Review, dev tools audit.

### 6.2 MCP and connectors

- [ ] .cursor/mcp.json (or config/mcp-servers) present; MCP servers you need are configured.
- [ ] Secrets (GITHUB_PERSONAL_ACCESS_TOKEN, FIRECRAWL_API_KEY, etc.) in 1Password or Cursor secrets; not in repo.
- [ ] New MCPs/connectors added only via config/canonical-mcp-connectors.yaml; sync script run after edit.
- [ ] verify-all-connections.sh run after connector changes (optional in Full Review).

### 6.3 Repos and sync

- [ ] config/repos.yaml has correct ROUTES and all_repos.
- [ ] After master file or governance change: route-constitution-to-repos.sh and sync-all-repos-and-review.sh run (or connect-and-configure-onewishos.sh).
- [ ] sync-dev-tools-repos-from-config.sh run after adding a repo so workspace and other dev tools see it.

### 6.4 Review and audit

- [ ] Before commit: run-full-review.sh run; mandatory checklist in full-review-summary.md completed.
- [ ] Commands-and-architecture review runs (Phase 3 of Full Review, step 6 of Comprehensive Eval); commands-and-architecture-review-latest.md checked for scripts to add to index.
- [ ] When doing full Review: run-review.sh or run-every-turn.sh; post-run eval and chain-of-events acknowledged.

### 6.5 Principles and rules

- [ ] Human-in-the-loop and claim typing applied to agent outputs where applicable.
- [ ] Simulation disclaimer included in generated litigation-related output.
- [ ] No secrets committed; .env.example only in repo.
- [ ] Destructive actions (1P execute, delete repos) only with explicit approval and dry-run first.

### 6.6 Dev tools parity

- [ ] No-fail dev tools audit (DEV_TOOLS_CANONICAL_LIST_AND_AUDIT §3) run when auditing or onboarding; every tool in §1 verified.
- [ ] Cursor, Claude Code, Codex, Copilot, v0, VS Code, Windsurf, Warp, Goose — each has constitution ref and entry point when in use.
