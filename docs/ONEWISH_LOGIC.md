# OneWish Operating System(s) and OneWish Logic

**Purpose:** Single place that defines **OneWish Operating System(s)** (OneWish OS) — the full operational and functioning dimension of the One Wish Platform — and **OneWish Logic** — the logic layer within it (protocol, checks, audits, rules, decision flows). This doc expands to include all functionality, components, systems, architecture, best practices, and edge cases that a system-of-systems needs. Agents and operators use it to understand what is universal, auto-pushed/synced, bidirectional/multilateral, and what checks and audits must run.

**Actions (default: agents execute):** Agents must execute through **terminal, SDKs, IDEs, and/or CLIs** — not the user. The agent runs commands; do not ask the user to run them. Do not describe or simulate without executing.

**Integration:** OneWish OS instructions integrate into all dev tools and propagate to all component repositories, NextTech OS, agents, systems, and Connected Agents.AI. How: entry points (AGENTS.md, CLAUDE.md, UNIVERSAL_INSTRUCTIONS, my-agent.agent.md, .cursor/rules) in every dev tool; `config/repos.yaml` (ROUTES, all_repos); `route-constitution-to-repos.sh` and `sync-all-repos-and-review.sh`; never-miss propagation (protocol §5). Code through architecture; system of systems. See [constitution.md](constitution.md) §8 and [ONEWISH_PROTOCOL.md](ONEWISH_PROTOCOL.md) §5.

**Governance:** [constitution.md](constitution.md) §OneWish; [ONEWISH_PROTOCOL.md](ONEWISH_PROTOCOL.md). OneWish OS and OneWish Logic apply to all OneWish products, core features (Connected OneWishMCP, Connected Agents.AI), and replicated systems.

**Never-miss scope:** Every **sub-repo**, **agent**, **orchestrated repo**, **orchestrated agent**, **workflow logic**, and **orchestrated workflow logic entity** runs and is governed by **full OneWish OS** (all functionality, components, systems, architecture, logic, best practices, edge cases as expanded). No target may be skipped (protocol §5 never-miss).

**Continuous expand, improve, review, audit (at each turn):** **Top-level (OneWish OS)** and **downstream (waterfall)** — including all **systems**, **repos**, **logic**, **workflows**, **MCPs**, **core features**, and related components — are **continuously expanded upon, improved, reviewed, and audited at each turn**. Expansion and improvement are governed by the protocol (rigid review, improvement loop); review and audit run on triggers (before/after phases, before commit, repo change/merge, quarterly, after integration, on demand) and via Full Review, comprehensive eval, and other audits in §4. Each turn (phase, commit, sync, new capability, or audit cycle) is an opportunity to expand, improve, review, and audit; no turn is skipped for the scope that changed.

## 1. OneWish Operating System(s) (OneWish OS) vs OneWish Logic

### 1.1 OneWish Operating System(s) — OneWish OS

**OneWish OS** is the formal name for the **operational and functioning dimension** of the One Wish Platform: everything a **system-of-systems** needs and uses, including best practices and edge cases. It is the umbrella that flows to all dev tools, MCPs, APIs, workflows, repos, agents, and future systems (e.g. OneWish Ollama, Claudebot, Web agents, OpenClaw instances) with no target missed. **Positioning:** OneWish OS is the **only** multi system-of-systems and architecture with sub-OS architecture and functionality that is **simple and unbreakable**, broken down to **very simple lanes**, segmented into **sub-OS master repos or GitHub accounts** (separate repos per section, informed by the sub-OS), and **connected with bidirectional and/or multilateral sync and/or push, automatically across all OS repos**. Sub-OS updates flow to **ConnectedAgents.AI OS Foundry** and back, integrating **another vector of virtuous and recursive improvement**. See [GAME_CHANGERS.md](GAME_CHANGERS.md) and [ONEWISHOS_CRYSTAL_CLEAR_ARCHITECTURE_AND_AGENT_TAXONOMY.md](ONEWISHOS_CRYSTAL_CLEAR_ARCHITECTURE_AND_AGENT_TAXONOMY.md).

**Inclusive scope (expand):** OneWish OS **includes all** of the following, and any like elements: **sub-OS** (every child operating system), **repos** (master, sub-repos, ROUTES, all_repos), **workflows**, **architectures**, **tools**, **functions**, **components**, **modules**, **MCPs**, **artifacts**, **methods**, **strategies**, **goals**, **agents**, and related governance, best practices, and edge cases. **Each sub-OS** (e.g. LitigationForce.AI, Financial Audit & Tax, RobertBaileyFamilyOffice) **includes in turn all sub-repo OS** for its scope: that is, each sub-OS contains the full set — its own repos, workflows, architectures, tools, functions, components, modules, MCPs, artifacts, methods, strategies, goals, agents, etc. — so the inclusion is recursive and complete at every level.

### 1.2 OneWish Logic (the logic layer)

**OneWish Logic** is the **logic** component within OneWish OS: the codified combination of protocol, checks, and audits (rules and decision flows) that keeps the platform aligned.

| Pillar                            | Contents                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Protocol**                      | [ONEWISH_PROTOCOL.md](ONEWISH_PROTOCOL.md) — architecture, sync, rigid review (usability, operability, efficiency), improvement (constant improvement loop), **never-miss propagation** (§5), **special projects** (§6).                                                                                                                                                                                                                                                                            |
| **Checks**                        | Verification and review steps that must run so the platform stays aligned (see §3).                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Audits**                        | Periodic and trigger-based audits that validate the architecture and produce actionable output (see §4).                                                                                                                                                                                                                                                                                                                                                                                            |
| **Feature and deliverable logic** | [ONEWISH_FEATURE_AND_DELIVERABLE_LOGIC.md](ONEWISH_FEATURE_AND_DELIVERABLE_LOGIC.md) — **primary deliverable** for a branch: documents (whitepaper, constitution, branding guide, rules) vs working agent/orchestration code (BrandingGuideAgent, RecommendationEngine, factory workflow) vs both. Four modes: (1) Documents first, (2) Agents + documents together, (3) Agent code only, (4) Other/Skip. Defines how **relevant features** are delivered and how they align with sync and wording. |

Together with the rest of OneWish OS (functionality, components, systems, architecture, best practices, edge cases), OneWish Logic ensures the operational and functioning dimension flows to all targets with no target missed.
